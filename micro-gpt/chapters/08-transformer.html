The Transformer Block

Advanced   How attention, the MLP, and residual connections snap together into one repeatable unit.

Where we are. Chapter 6 explained the MLP and the normalization/activation layers. Chapter 7 explained attention. This chapter assembles those pieces into the transformer block — the single unit that GPT stacks over and over. Then we follow one character all the way from input to a predicted next character.

1. The block diagram

A transformer block takes a 16-number vector in and produces a 16-number vector out. In between, it runs attention and an MLP, each wrapped with a normalization step and a residual connection (the "save it and add it back" arrows). Here's the whole shape:

Input — 16-dim vector
↘ save a copy as residual #1
RMSNorm
Multi-Head Attention  ←  keys, values from all past positions
+ add residual #1 back
↘ save a copy as residual #2
RMSNorm
MLP: Linear → ReLU → Linear  (16 → 64 → 16)
+ add residual #2 back
Output — 16-dim vector

Two halves, each built the same way:

Attention half

save input → RMSNorm → attention → add the saved input back. The model asks "what in my past is relevant?" and folds the answer into the stream.

MLP half

save again → RMSNorm → MLP → add the saved value back. The model does per-position "thinking" on the result, again folding it in.

The pattern to memorize: x = x + sublayer(norm(x)). Normalize a copy, run a sublayer on it, add the result back onto the original. Attention and the MLP are both wrapped in exactly this. Everything else is detail.
But why normalize before attention and the MLP, and not after? This is called "pre-norm" (the original 2017 Transformer used "post-norm," normalizing after each sublayer). Pre-norm is far more stable to train: every sublayer receives an input at a consistent scale, and the residual stream stays clean because the norm only touches the copy that goes into the sublayer. Post-norm tends to blow up or vanish in deep stacks. That's why GPT-2 and essentially every modern model use pre-norm — exactly the x = x + sublayer(norm(x)) shape above.
But why this exact order: norm → attention → add → norm → MLP → add? The two sublayers do complementary jobs. Attention lets positions communicate — it mixes information across the sequence ("what earlier characters are relevant to me?"). The MLP does the opposite — it processes each position on its own, transforming what that position just gathered. Alternating them gives the model a "look around, then think" rhythm. The norms keep the scale sane and the residual adds keep the gradient flowing, so the two can be stacked indefinitely.
Real-life analogy A transformer block is one cycle of a classroom discussion. Step 1 (Attention): "Look around — what have the other students said?" Step 2 (Residual): "Combine what I just heard with what I already knew." Step 3 (MLP): "Now think privately about what I learned." Step 4 (Residual): "Combine that fresh thinking with my previous understanding." Stacking layers just repeats the cycle, each round deepening everyone's grasp.

2. Walking through the gpt() function

Here is the forward pass from the Java code. It glues together the embedding lookup, the transformer block, and the final projection to logits. Markers below. 

Value[] gpt(int tokenId, int posId, List<Value[]>[] keys, List<Value[]>[] values) {
    // (1) token embedding + position embedding
    Value[] x = new Value[nEmbd];
    for (int i = 0; i < nEmbd; i++)
        x[i] = sd.get("wte")[tokenId][i].add(sd.get("wpe")[posId][i]);

    // (2) initial RMSNorm (no gain parameters — just rescale)
    x = rmsnorm(x);

    for (int li = 0; li < nLayer; li++) {     // nLayer = 1 here
        // (3) save x, norm, then attention block  (see Chapter 7)
        Value[] xr = x;
        x = rmsnorm(x);
        // ... Q, K, V projections, multi-head attention ...
        // attention output projected through attn_wo

        // (4) first residual: add attention output back onto saved x
        for (int i = 0; i < nEmbd; i++) x[i] = x[i].add(xr[i]);

        // (5) save x, norm, then MLP: Linear -> ReLU -> Linear
        xr = x;
        x = rmsnorm(x);
        x = linear(x, sd.get("layer" + li + ".mlp_fc1"));
        for (int i = 0; i < x.length; i++) x[i] = x[i].relu();
        x = linear(x, sd.get("layer" + li + ".mlp_fc2"));

        // (6) second residual: add MLP output back onto saved x
        for (int i = 0; i < nEmbd; i++) x[i] = x[i].add(xr[i]);
    }

    // (7) project to logits: 16-dim vector -> 27 scores (no final norm)
    return linear(x, sd.get("lm_head"));
}
1 Embed and place. Look up the character's meaning vector and add a "where am I in the sequence" vector. (Detailed below.)
2 Initial RMSNorm. Rescale the vector so the numbers are a sane size. Note: rmsnorm() in this code has no learnable gain parameter — it just normalizes by root-mean-square.
3 Save + norm + attention. Save x for the residual, normalize a fresh copy, then run the Chapter 7 attention mechanism.
4 Residual #1. Add attention's output back onto the saved input. The block adjusts the stream rather than replacing it.
5 Save + norm + MLP. Same pattern: save, normalize, then run the two-layer MLP with a ReLU in the middle (16 → 64 → 16).
6 Residual #2. Add the MLP's output back. Same additive idea.
7 Project to logits. A linear layer turns the 16 numbers into 27 scores — one per possible next character. Unlike standard GPT-2, there is no final normalization here.

3. The full forward pass — following the letter 'c'

Let's trace one real prediction. We feed the letter c at position 0 and watch what happens, step by step.

#StepWhat the data looks like
1Look up token embedding for ca vector of 16 numbers (its learned "meaning")
2Look up position embedding for position 0another 16 numbers ("I'm at the start")
3Add them together16 numbers — meaning + place, combined
4RMSNorm → Attention → add residual16 numbers, now informed by the (empty so far) past
5RMSNorm → MLP → add residual16 numbers, refined by per-position "thinking"
6Final RMSNorm + linear projection27 numbers (logits) — one score per vocab character
7Softmax over the 27 logits27 probabilities that sum to 1
8Pick the highest (or sample)one character — the model's guess for what follows c
Why 27? The model is trained on lowercase names: 26 letters plus one special "end of name" marker = 27 possible characters. So the vocabulary size is 27, and the final layer always outputs exactly 27 scores. After softmax, the largest probability tells you the most likely next character.
But why does the final projection (lm_head) go from 16 to 27? Inside the block, the model thinks in a 16-dimensional vector — a compact "thinking space" with no direct meaning per slot. But to actually answer, it must pick among 27 possible next characters. lm_head is the bridge: a linear layer that converts the 16-number thinking vector into one score per option (27 of them, the "decision space"). Softmax then turns those 27 scores into probabilities. The 16 is internal capacity; the 27 is fixed by the vocabulary.
At position 0 there's nothing to attend to but itself. When c is the very first character, the attention step has only one position in its key/value lists — position 0. So attention can only "blend" c with itself. Its real power shows up later in the sequence, once there's a past to look back on. That's exactly the causal growth from Chapter 7.
Concrete example

Let's push one real letter through the whole block. We're at the letter o in john (sequence so far: BOS, j, o), so the token is o (id 14) at position 2. Watch the shape of the data at every stage:

Input: token id 14, position 2.

  1. Embedding: wte[14] + wpe[2] → 16 numbers (meaning + place).
  2. RMSNorm: rescale → 16 numbers at a sane magnitude.
  3. Attention: project to Q, K, V; 4 heads each output 4 numbers; concatenate → 16; output projection → 16.
  4. Residual #1: add the attention output back onto the pre-attention vector → 16.
  5. RMSNorm again → 16.
  6. MLP: 16 → 64 (expand) → ReLU → 64 → 16 (compress) → 16.
  7. Residual #2: add the MLP output back → 16.
  8. lm_head: 16 → 27 (one score per possible next character).

Output: the 27 logits, then softmax. Illustrative numbers for the two that matter here:
logits = […, 'h':3.2, …, 'n':2.1, …] → softmax → […, 'h':0.35, …, 'n':0.12, …]

The model puts the most probability on h — which is correct, since john continues o → h. Every number from step 1 to step 7 was a 16-vector; only the final lm_head widens it to 27, the size of the vocabulary.

4. Why "layers"?

Our gpt() loop runs nLayer = 1 — a single transformer block. That's why the whole model is so small. The only thing GPT-4 changes here is the count:

microGPT

  • nLayer = 1
  • one block, run once
  • learns spelling-ish patterns for names

GPT-4 (roughly)

  • ~120 layers
  • the same block, stacked 120 times
  • early layers learn spelling, middle layers grammar, later layers meaning

Stacking just means making the loop run more times: the output of block 1 becomes the input of block 2, and so on. Each block has its own learned weights, so each can specialize. More layers = more chances to transform the vector into something richer before it becomes a prediction.

But why only 1 layer in our model? Because the task is genuinely simple: character-level name generation over a 27-character vocabulary. One block is enough to learn patterns like "after sh, an a is common" or "names rarely end in q." Real language needs depth: early layers learn spelling and grammar, middle layers learn meaning, late layers learn task-specific behavior — and you can only build that hierarchy by stacking many blocks. GPT-4 uses ~120. We use 1 because one is all the problem demands.
Same block, more reps. There is no architectural difference between microGPT and GPT-4's blocks — only how many are chained, how wide the vectors are, and how much data they trained on. You're looking at the real recipe.

5. The "residual stream" view

Here's the mental model that makes transformers click. Picture the 16-number vector as a stream of information flowing straight down the page from input to output. Each sublayer (attention, then MLP) doesn't replace the stream — it reads from it and writes a small correction back onto it.

x  =  x  +  attention(norm(x))
x  =  x  +  mlp(norm(x))

Because the update is always an addition (the + x), information never gets destroyed — it can only be nudged. Attention contributes "here's relevant context from the past." The MLP contributes "here's some processing of that." Both get added into the same running stream.

Real-life analogy Residual connections are like taking notes during a lecture. You listen to the lecture (attention + MLP) and jot down new understanding — but you also keep your original notes from before. The residual connection adds the new understanding to the old, so even if the lecture was confusing and the block learned nothing useful, you haven't lost what you already knew.
Why additive updates help. Two reasons. First, training: gradients (Chapter 9) flow cleanly backward through a plain addition, so even a deep 120-block stack can learn without the signal fading away. Second, behavior: each block can make a tiny, targeted edit to the stream instead of having to reconstruct everything from scratch. Small additive corrections, stacked many times, add up to sophisticated behavior.
