Training & Optimization

Advanced How a pile of 4,192 random numbers becomes a name generator. We measure wrongness, figure out which way to nudge every number, and take a small step — a thousand times.

By now you know the model is just parameters (Chapter 1) and that a forward pass turns tokens into a guess (Chapters 6–8). This chapter is the other half: learning. Learning is a loop of four moves — guess, measure how wrong, find the downhill direction, step. Let's build each move from the ground up.

1. The loss function — turning "wrong" into a number

The model's output for one position is a probability for every one of the 27 possible next tokens (that's what softmax gave us in Chapter 4). We know the real next character from the training name. So we can ask one sharp question: how much probability did the model put on the correct answer?

If it put 100% on the right letter, it was perfect. If it put almost nothing there, it was badly wrong. We need to convert "probability of the correct answer" into a single loss number where small means good. The formula that does this is called cross-entropy loss, and for a single prediction it is dead simple:

loss = −log( probability of the correct token )

That minus-log is the whole trick. Remember from Chapter 4 that log() of a number between 0 and 1 is negative, and it dives toward −∞ as the number approaches 0. Sticking a minus sign in front flips it positive, so:

  • Correct token got probability 0.7 → loss = −log(0.7) = 0.36. Pretty good.
  • Correct token got probability 0.01 → loss = −log(0.01) = 4.6. Much worse!
  • Correct token got probability 1.0 → loss = −log(1) = 0. Perfect, zero loss.
Why log, and not just "1 minus the probability"? Log punishes confident mistakes brutally. Being 1% sure of the right answer isn't just "a bit worse" than 70% — it's twelve times the loss. That steep penalty pushes the model to stop being confidently wrong, which turns out to be exactly the right pressure. All logs here are natural logs (base e), matching Java's Math.log.
Real-life analogy Cross-entropy is a teacher grading confidence. A student says "I'm 90% sure the answer is A." If it really is A, the teacher gives a great grade (low loss). If it's actually B, the teacher gives a terrible grade (high loss) — because the student was confident and wrong. The more confident a wrong answer, the harsher the penalty. That's exactly why cross-entropy punishes confident mistakes far more than uncertain ones.
But why cross-entropy, and not just "percentage correct"? Because "percentage correct" isn't differentiable. It jumps between 0 and 1 with a flat line in between, so its gradient is zero almost everywhere — useless for gradient descent, which only knows how to follow slopes. Cross-entropy is smooth: it gives partial credit for being almost right (probability 0.9 on the correct answer) and punishes being confidently wrong (0.01 on the correct answer) far more steeply. That continuous gradient is what lets the model improve in tiny, learnable steps.

In the Java code, this is one line inside the training loop:

            Value loss = Value.of(0);
            for (int pos = 0; pos < n; pos++) {
                Value[] logits = gpt(tokens.get(pos),
                                     pos, keys, vals);
                Value[] probs = softmax(logits);
                loss = loss.add(
                    probs[tokens.get(pos + 1)]
                        .log().neg());
            }
            loss = loss.div(Value.of(n));

Read probs[tokens.get(pos + 1)] as "the probability the model assigned to the token that actually comes next." We take its .log(), then .neg() (negate it), and add it onto the running loss. After looping over every position in the name, we divide by n to get the average loss per character. One number summarising how wrong the model was on this whole name.

But why average the loss over the sequence (divide by n)? Without averaging, a longer name would rack up a bigger total loss just because it has more characters — more positions, more chances to be penalized. The model would then treat long names as "more important" and preferentially learn short ones, purely as an artifact of length. Dividing by n makes the loss a per-character average, comparable across names of any length, so every name pulls on the weights with equal weight.
The "random guessing" baseline With 27 equally-likely tokens, a clueless model puts 1/27 on each. Its loss is −log(1/27) = log(27) ≈ 3.30. That's why untrained loss starts around 3.3–3.4: the model is literally guessing. Training's job is to drive that number down.

2. Backpropagation — which way is downhill?

We have one number, loss, and we want it smaller. The loss was produced by a long chain of arithmetic involving all 4,192 parameters. The question is: for each parameter, if I wiggle it slightly, does the loss go up or down, and how fast?

That "how fast does loss change when I wiggle this one parameter" number is the parameter's gradient. Computing all 4,192 gradients by hand would be insane. Instead we call one method — the engine you built in Chapter 5: 

            // Backward pass for gradient computation
            loss.backward();

A single call to loss.backward() walks the entire computation graph backwards, applies the chain rule at every node, and fills in the .grad field of every parameter. After it returns, each parameter knows its own gradient.

What a gradient actually tells you A parameter's gradient is a slope: "if I increase this parameter by a tiny amount h, the loss changes by roughly grad × h."
  • Positive gradient → increasing the parameter increases loss (bad). So we should decrease it.
  • Negative gradient → increasing the parameter decreases loss (good). So we should increase it.
Either way: move the parameter in the direction opposite to its gradient. That single rule is the heart of all the training that follows.

3. Gradient descent — taking the step

Now we use those gradients. The simplest possible update, called gradient descent, is one line per parameter: 

param.data -= learningRate * param.grad;

Subtracting grad moves us opposite to the gradient — downhill, toward lower loss. The learning rate controls how big a step we take. In our code it starts at 0.01. Picking it is a balancing act:

Too small

Each step barely moves. Training crawls; you'd need millions of steps to get anywhere. Safe but painfully slow.

Just right

Steps are big enough to make real progress each time, small enough not to fly past the bottom. Loss falls steadily.

Too big

You leap clear over the valley to a higher spot on the other side, then leap back even higher. Loss oscillates or explodes to infinity.

The marble-in-a-bowl picture Imagine loss as a landscape and the parameters as the position of a marble. The gradient is the slope under the marble. Gradient descent rolls the marble a little way downhill, re-checks the slope, rolls again. Learning rate is how far you shove it each time.

4. Adam — a smarter step (what our code really uses)

Plain gradient descent uses the same learning rate for every parameter and forgets everything between steps. That's wasteful. Some parameters have steep, consistent slopes; others have tiny, noisy ones. They shouldn't all take identically-sized steps. The Adam optimizer fixes this by remembering two running averages for each parameter i:

  • m[i] — momentum. A smoothed average of recent gradients. If a parameter keeps getting pushed the same way, momentum builds up and it moves faster in that direction — like a ball that keeps rolling.
  • v[i] — squared gradient. A running average of how large this parameter's gradients have been. It measures how "noisy" or "active" the parameter is.
Real-life analogy Adam is a ball rolling downhill with momentum. Plain gradient descent cautiously steps downhill one tiny step at a time; Adam is a ball that builds up speed — rolling faster in consistent directions and slowing down on bumpy terrain. It also adjusts its speed per-dimension: if the slope has been steadily steep one way, the ball learns to roll faster there.
But why Adam and not plain gradient descent? Plain gradient descent uses the same learning rate for all 4,192 parameters. But they don't all need the same step: some sit in flat regions and need a big shove, others sit near a sharp optimum and a big step would overshoot. Adam adapts the step size per parameter, using running averages of each one's past gradients — effectively giving every parameter its own speedometer. That per-parameter tuning is why it converges faster and more reliably than one global learning rate ever could.

Here is the exact Adam update from our training loop, line by line:

            // Adam parameter update with linear LR decay
            double lrt = lr * (1.0 - step / (double) nSteps);
            for (int i = 0; i < params.size(); i++) {
                double g = params.get(i).g();
                m[i] = b1 * m[i] + (1 - b1) * g;
                v[i] = b2 * v[i] + (1 - b2) * g * g;
                double mh = m[i]
                    / (1 - Math.pow(b1, step + 1)),
                    vh = v[i]
                    / (1 - Math.pow(b2, step + 1));
                params.get(i).data[0] -=
                    lrt * mh / (Math.sqrt(vh) + epsAdam);
                params.get(i).grad[0] = 0;
            }
Line by line:
  • g = params.get(i).g() — grab this parameter's gradient (set by backward()).
  • m[i] = b1*m[i] + (1-b1)*g — blend 85% of the old momentum with 15% of the new gradient. Smooths out jitter so a single weird gradient doesn't yank us around.
  • v[i] = b2*v[i] + (1-b2)*g*g — same idea for the squared gradient. g*g is always positive, so this tracks raw magnitude.
  • mh = m[i] / (1 - pow(b1, step+1)) and vh similarly — bias correction. Both averages start at 0, so early on they read too low; this division inflates them back to honest values during the first steps.
  • param.data -= lrt * mh / (sqrt(vh) + eps) — the actual step. Notice we divide the smoothed gradient mh by sqrt(vh). A parameter with big noisy gradients (large vh) gets a smaller effective step; a calm parameter gets a larger one. eps (1e-8) just prevents divide-by-zero.
But why beta1 = 0.85 and beta2 = 0.99? beta1 controls momentum — how much past gradients steer the current step. 0.85 means "keep 85% of the previous direction, mix in 15% of the new gradient," smoothing out jitter. beta2 controls the adaptive rate — how long a memory of gradient magnitudes to keep. 0.99 means "average over a long history of how big this parameter's gradients have been." Both are close to the Adam paper's defaults (0.9 and 0.999), nudged slightly for this small, fast-training model.
The ball-on-a-hill analogy, completed 🏀 Momentum = the ball keeps rolling the way it was already going, so it powers through small bumps and flat patches. Adaptive rate = the ball instinctively slows down on steep, treacherous slopes and speeds up across gentle ones. Adam is gradient descent with both of those instincts bolted on — which is why it's the default optimizer almost everywhere.
Concrete example

Input: one parameter — say weight p[42] somewhere in the attention layer.

  • Current value: 0.05
  • Its gradient this step: -0.003 (negative → loss decreases if we increase this weight, so we should nudge it up)
  • Learning rate at step 500 of 1000: 0.01 × (1 - 500/1000) = 0.005

Output: the plain gradient-descent step is p -= lr × grad = 0.05 - 0.005 × (-0.003) = 0.050015 — an upward nudge of 0.000015. (Adam rescales the magnitude by normalizing the gradient against its own running size, but the direction is the same: up.)

One weight barely twitches. But there are 4,192 of them, and the loop runs 1,000 steps — about 4.2 million tiny adjustments. That accumulation is what turns a pile of random numbers into a name generator.

5. Learning-rate schedule — big steps first, fine steps last

There's one more refinement. Early in training the model is terrible and far from any good answer — big steps are great, they cover ground fast. Late in training it's nearly right and just needs gentle polishing — big steps would knock it back off target. So we shrink the learning rate as training proceeds. Our code does a simple linear decay: 

            double lrt = lr * (1.0 - step / (double) nSteps);

At step = 0 the factor (1.0 - 0/1000) is 1, so lrt = 0.01 — full size. At the final step the factor approaches 0, so lrt fades to nearly 0 — tiny, careful nudges. Smooth ramp from coarse to fine over the whole run.

Real-life analogy Learning-rate decay is landing a plane. At the start the pilot descends rapidly (high learning rate — big parameter updates). As the runway approaches, the descent gets gentler (lower learning rate — smaller, more precise adjustments). You don't want to slam into the runway at full speed — that's overshooting the optimal parameters.
But why linear decay — why shrink the learning rate at all? At the start, the model is far from any good solution, so big steps are fine and even helpful — they cover ground fast. As training proceeds it gets close to a good answer, and big steps would overshoot and bounce back off target. Linear decay smoothly trades one for the other: fast and coarse early, slow and precise late. It's like driving fast on the open highway, then easing off as you roll into the parking spot.

6. Putting it together — the complete training loop

Here is the entire learning process, exactly as it appears in main(). Everything above lives inside this one loop: 

        for (int step = 0; step < nSteps; step++) {
            String doc = docs.get(step % docs.size());
            List<Integer> tokens = new ArrayList<>();
            tokens.add(BOS);
            for (char c : doc.toCharArray())
                tokens.add(uchars.indexOf(c));
            tokens.add(BOS);

            // Forward pass building the computation graph
            int n = Math.min(blockSize,
                             tokens.size() - 1);
            List<Value[]>[] keys = new List[nLayer],
                            vals = new List[nLayer];
            for (int i = 0; i < nLayer; i++) {
                keys[i] = new ArrayList<>();
                vals[i] = new ArrayList<>();
            }

            Value loss = Value.of(0);
            for (int pos = 0; pos < n; pos++) {
                Value[] logits = gpt(tokens.get(pos),
                                     pos, keys, vals);
                Value[] probs = softmax(logits);
                loss = loss.add(
                    probs[tokens.get(pos + 1)]
                        .log().neg());
            }
            loss = loss.div(Value.of(n));

            // Backward pass for gradient computation
            loss.backward();

            // Adam parameter update with linear LR decay
            double lrt = lr * (1.0 - step / (double) nSteps);
            for (int i = 0; i < params.size(); i++) {
                double g = params.get(i).g();
                m[i] = b1 * m[i] + (1 - b1) * g;
                v[i] = b2 * v[i] + (1 - b2) * g * g;
                double mh = m[i]
                    / (1 - Math.pow(b1, step + 1)),
                    vh = v[i]
                    / (1 - Math.pow(b2, step + 1));
                params.get(i).data[0] -=
                    lrt * mh / (Math.sqrt(vh) + epsAdam);
                params.get(i).grad[0] = 0;
            }

            System.out.printf(
                "\rstep %4d/%4d | loss %.4f",
                step + 1, nSteps, loss.d());
        }
  1. Pick a name — docs.get(step % docs.size()) cycles through the dataset.
  2. Tokenize — wrap the characters in BOS markers and convert each letter to its integer id (Chapter 1).
  3. Forward pass — for every position, call gpt(...) to get logits, softmax them to probabilities, and accumulate the cross-entropy loss against the true next token.
  4. Backward pass — loss.backward() fills every .grad.
  5. Update — the Adam block steps all 4,192 parameters downhill, then zeroes their gradients ready for the next step.
But why reset the gradients to zero after each step (grad[0] = 0)? Because backward() accumulates gradients (it does +=, not =). If we never reset, step 2's gradients would still carry step 1's leftovers, step 3 would carry steps 1+2, and so on. The model would end up optimizing for a tangled mixture of all past examples at once instead of the current one. Zeroing after each update gives every step a clean slate — exactly the gradient for the name we just looked at.
What you actually see when it runs The printed loss starts near 3.4 (random guessing, ~log 27) and, over 1,000 steps, falls to roughly 2.2. That drop is the model genuinely discovering the statistical shape of human names — which letters follow which, how names tend to start and end — purely from being corrected toward real examples. No rule about names was ever written down. That fall is the learning.
Before you move on Four ideas to lock in: (1) loss = −log(prob of correct token), small is good; (2) backward() fills every gradient via the chain rule; (3) a gradient points uphill, so we step the opposite way; (4) Adam adds momentum and per-parameter step sizing, and a decaying learning rate goes coarse→fine.
