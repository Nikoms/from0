The Attention Mechanism

Advanced   Query, Key, Value — the one idea that makes transformers work.

This is the chapter that matters most. Everything before this — vectors, embeddings, softmax, neural network layers — was preparation. Attention is the mechanism the "T" in GPT (Transformer) is built around. If you understand this chapter, you understand the heart of ChatGPT. Take it slowly.

1. The core intuition: not everything matters equally

A GPT predicts the next character by looking at the characters that came before. But here's the key insight: when you're guessing the next letter, the previous letters are not all equally useful.

Imagine the model has seen these letters and must guess what comes next:

A   l   e   x   a   n   d   r   ?

What's the next letter? Probably a (to spell "Alexandra") or o ("Alexandro"). To make that guess well, which earlier letters should you pay attention to?

  • The capital A at the very start tells you this is a name and hints at the whole "Alexand-" family.
  • The recent r tells you you're near the end of the stem.
  • The l in the middle? Barely relevant to the next letter at all.

A naive model would mush all nine previous letters together with equal weight. Attention does something smarter: it lets the model choose, for each prediction, how much to focus on each previous position. Some get a lot of focus, some get almost none. Those focus amounts are learned during training.

One sentence definition. Attention is a weighted average of past information, where the model itself computes the weights based on what it's currently looking for.
Real-life analogy Attention is a cocktail party. You're in a noisy room (many positions in the sequence) and can't listen to everyone equally. So you decide who to focus on based on what you're looking for (your query) and what each person is saying (their key), then you take a weighted mix of what those people actually say (their value). Multi-head attention, later in this chapter, is like following four conversations at once — each tuned to a different topic.

2. Query, Key, Value — the QKV pattern

The mechanism has three pieces with intimidating names: Query, Key, and Value. The names come from databases and search engines, and the search-engine analogy is the fastest way in.

The YouTube search analogy

Think about what happens when you search YouTube for "funny cat videos":

Search conceptWhat it isIn attention
Query What you type into the search bar: "funny cat videos" What the current position is looking for
Key The title and tags of every video in the database What each past position advertises about itself
Value The actual video you watch The actual information each past position carries
Score How well your query matches a video's title/tags How much attention to pay to that position
Result A feed blended from the best-matching videos A blend of values, weighted by the scores

You type one query. It gets compared against every video's key. The videos whose keys match your query best get shown the most. The thing you actually receive is the content (the value) of those videos, mixed together according to how well they matched.

Attention does exactly this, but the "search" happens for every position in the sequence:

Query — "what am I looking for?"

Computed from the current position's vector. It encodes the question this position is asking of the past.

Key — "what do I contain?"

Computed from each past position's vector. It's that position advertising what it's about.

Value — "what info do I carry?"

Also computed from each past position. It's the actual content that gets passed forward if this position is attended to.

Where do Q, K, V come from?

Each is just the position's embedding vector multiplied by a learned weight matrix (a Linear layer from Chapter 6). Same input, three different transforms.

They're all derived from the same vector. A position starts as one 16-number embedding vector. We run it through three separate linear layers to get its query, its key, and its value. The model learns those three weight matrices so that the resulting queries and keys line up usefully.
But why Query, Key, Value — why not just compare the embeddings directly? A raw embedding encodes "what this character is." But attention needs two more things: what a position is looking for (its query) and what it offers to others (its key) — and those are not the same as its identity. The three linear layers (wq, wk, wv) learn those separate roles. The same character a might look for completely different things depending on context, which a single fixed embedding could never express on its own.

3. Step by step, with real numbers

Let's work a complete example by hand using the three-letter sequence "cat". Three positions: c (position 0), a (position 1), t (position 2).

In the real code each position is a 16-dimensional vector. Sixteen numbers per step is too many to follow on paper, so for this worked example we'll pretend each position is only 4 numbers. The logic is identical; only the count changes.

Step A — start with the embedded positions

After embedding (Chapter 3), suppose our three letters became these vectors:

c = [1.0,  0.0,  1.0,  0.0]
a = [0.0,  1.0,  0.0,  1.0]
t = [1.0,  1.0,  0.0,  0.0]

Step B — make Q, K, V with linear transforms

We pass each vector through three learned weight matrices. The actual numbers in those matrices were learned during training; here we'll just show the result of applying them. Each letter now has a query, a key, and a value (still 4 numbers each):

PositionQuery (Q)Key (K)Value (V)
c[1.0, 0.2, 0.1, 0.0][0.9, 0.1, 0.2, 0.0][2.0, 0.0, 1.0, 0.5]
a[0.1, 1.0, 0.0, 0.3][0.2, 0.9, 0.1, 0.4][0.0, 3.0, 0.0, 1.0]
t[0.5, 0.5, 0.2, 0.1][0.4, 0.6, 0.0, 0.2][1.0, 1.0, 2.0, 0.0]

Step C — score every query against every key (dot product)

The match between a query and a key is their dot product: multiply matching slots, add them up. A bigger number means a better match. (We covered the dot product in Chapter 2 — it's large when two vectors point the same way.)

For example, position t's query against position c's key:

Qt · Kc = (0.5×0.9) + (0.5×0.1) + (0.2×0.2) + (0.1×0.0) = 0.54

Do that for every (query, key) pair and you get a 3×3 grid of raw scores. Row = the position doing the asking; column = the position being asked about:

query ↓ / key →cat
c0.940.390.52
a0.191.040.70
t0.540.610.52

Step D — divide by √(headDim)

Before going further we divide every score by the square root of the vector size. Here the head size is 4, and √4 = 2, so every number gets divided by 2.

Why divide at all? Dot products grow as vectors get longer. With 16-number vectors the raw scores can get large, and large numbers fed into softmax (next step) make it pick one winner almost 100% of the time — a too-confident, "spiky" distribution that's hard to train. Dividing by √(headDim) keeps the scores in a calm range so the model can spread its attention smoothly. This single trick is called "scaled" dot-product attention.
But why √(headDim) specifically, and not just any constant? The dot product of two random vectors grows in proportion to their length, so its spread (standard deviation) grows like √(headDim). Dividing by exactly √(headDim) cancels that growth, keeping the variance of the scores around 1 no matter how big the head is. With our tiny headDim = 4 it barely matters. But at GPT-2's headDim = 64 the un-scaled scores would be roughly 16× larger — enough to push softmax into an extreme, near one-hot spike where the gradient vanishes and learning stalls.

After dividing the row for query t by 2: [0.27, 0.31, 0.26].

Step E — softmax each row into weights that sum to 1

Raw scores aren't percentages. Softmax (Chapter 4) turns each row into positive numbers that add up to exactly 1 — actual attention weights. Applying softmax to query t's scaled row [0.27, 0.35, 0.30]:

softmax([0.27, 0.31, 0.26]) = [0.33,  0.34,  0.33]

This says: "When position t makes its prediction, it pays 33% attention to c, 34% to a, and 33% to itself." Every row of the grid becomes a set of weights like this.

Step F — blend the values using those weights

Finally we take a weighted average of the value vectors. For position t, with weights [0.33, 0.34, 0.33]:

outt = 0.33×Vc + 0.34×Va + 0.33×Vt
= 0.33×[2,0,1,0.5] + 0.34×[0,3,0,1] + 0.33×[1,1,2,0]
= [0.99,  1.35,  0.99,  0.51]

That output vector is what attention produces for position t: a custom blend of the past, mixed according to what t was looking for. Positions c and a get their own blends the same way.

Recap of the whole pipeline. embed → make Q, K, V → score (Q·K) → scale (÷√d) → softmax → weighted sum of V. Six steps. That's all attention is.

4. The causal mask — why we only look backward

You may have noticed something in the grid above: position c (the first letter) should not be allowed to look at a or t, because those come later. When the model is predicting what follows c, the letters a and t haven't happened yet. Letting it peek would be cheating — it would just copy the answer.

GPT is autoregressive: it generates one character at a time, left to right, and each prediction may only use the past. So the real attention grid is triangular: every position can attend to itself and everything before it, but nothing after.

query ↓ / key →cat
c1.00——
a0.400.60—
t0.330.340.33

Position c can only attend to itself, so its weight is forced to 1.00. Position a attends to c and a. Only t, the last position, sees all three.

Real-life analogy Causal masking is a "no spoilers" rule. Imagine reading a mystery novel where you're only allowed to see the pages you've already read — you can't flip ahead to find out whodunit. Masking enforces exactly this: position 3 can only attend to positions 0, 1, and 2, never to position 4 or beyond. During generation those future positions don't even exist yet.
But why force attention to look only backward at all? Because GPT generates text left to right: when it predicts position 5, positions 6, 7, 8 don't exist yet. If we let the model peek ahead during training, it would learn to lean on information it will never have at generation time — so we make training match the way the model is actually used. (BERT does the opposite: it reads the whole sentence at once with bidirectional attention, because it isn't generating text left to right — it's filling in blanks, so seeing the future is allowed there.)
How the Java code enforces this — for free. microGPT doesn't build a fancy mask matrix. It processes tokens one at a time and appends each new key and value to growing lists. When it computes attention for the token at position li, the key and value lists only contain entries up to li — the future literally doesn't exist yet in the lists. Look for this line in the code: 
// As we process each token, we append its key/value to per-layer lists.
// The lists only ever hold the current token and earlier ones —
// so attention can never reach a token that hasn't been added yet.
keys[li].add(k);    // k = this token's key  vector
values[li].add(v);  // v = this token's value vector

Because the lists grow as generation proceeds, the causal "can only look backward" rule is automatic. No mask needed.

5. Multi-head attention

So far we described one attention computation. Real transformers run several in parallel and call each one a head. microGPT uses nHead = 4 heads.

The embedding is 16 numbers wide (nEmbd = 16). We split that into 4 heads, giving each head headDim = 16 / 4 = 4 numbers to work with. Each head does the full QKV-score-softmax-blend dance on its own little 4-number slice.

Why split instead of one big head? Different heads can learn to focus on different things. With names, one head might learn to track "where did the last vowel appear?", another "is this the start of the word?", another "what consonant came just before?". One monolithic attention would have to cram all those jobs into a single set of weights. Four small heads can specialize.
But why is 4 heads of size 4 more expressive than 1 head of size 16? Because each head gets its own query/key/value projection, so each can chase a different pattern at the same time — one attending to the previous character, one to vowels, one to the first letter. A single size-16 head has only one set of Q/K/V weights, and therefore one "perspective": it has to average all those jobs into a single attention pattern. Splitting the same 16 dimensions into four independent views costs no extra parameters but buys genuine parallel specialization.
Concrete example

Suppose we're processing the letter h in the name john — the tokens seen so far are BOS, j, o, and now h. We'll watch head 0 only.

Input: h's vector after embedding + RMSNorm is 16 numbers, e.g.
x = [0.12, -0.05, 0.08, 0.21, | -0.11, 0.03, 0.17, -0.09, | 0.06, 0.14, -0.02, 0.10, | 0.19, -0.07, 0.04, 0.13]

Project to a query. Multiply x by wq to get a 16-number query, then slice it into 4 heads of 4:
q = [0.3, -0.1, 0.2, 0.4, | -0.15, 0.25, 0.1, -0.3, | 0.05, 0.18, -0.22, 0.09, | 0.11, -0.04, 0.27, 0.16]
Head 0 takes the first slice: q₀ = [0.3, -0.1, 0.2, 0.4]. That's the question this head is asking of the past.

Score against each earlier key (head 0's slice of each stored key):
k_BOS = [0.1, 0.2, -0.1, 0.3], k_j = [0.4, -0.2, 0.1, 0.1], k_o = [-0.1, 0.3, 0.2, -0.2]
Dot products: q₀·k_BOS = 0.03-0.02-0.02+0.12 = 0.11, q₀·k_j = 0.12+0.02+0.02+0.04 = 0.20, q₀·k_o = -0.03-0.03+0.04-0.08 = -0.10
Raw scores: [0.11, 0.20, -0.10].

Scale by ÷√4 = ÷2 → [0.055, 0.10, -0.05], then softmax → [0.34, 0.36, 0.31] (sums to ~1 after rounding).

Output: blend the value vectors with those weights.
v_BOS = [0.2, -0.1, 0.3, 0.1], v_j = [0.1, 0.4, -0.2, 0.3], v_o = [-0.3, 0.2, 0.1, -0.1]
out = 0.34×v_BOS + 0.36×v_j + 0.31×v_o ≈ [0.01, 0.17, 0.06, 0.11]

Those four numbers are head 0's summary of "what the sequence so far means for predicting what follows h." Here the weights are nearly even (0.34/0.36/0.31), so this head is spreading its attention across all three earlier tokens rather than fixating on one. The other 3 heads run the same six steps on their own slices; concatenating all four gives the 16-number result.

Simplified for clarity. The code below is a pedagogical version that's easier to read than the actual Java port. The real code computes Q, K, V for the full 16-number vector in one shot, then slices per head using offset indexing (Arrays.copyOfRange). It also stores keys/values per layer (not per head), accessing each head's slice via an hs offset. The math is identical — only the data plumbing differs. Chapter 10 has the exact source code.

Here's the multi-head loop, annotated (simplified from the actual code for readability):

// nHead = 4, headDim = 4, nEmbd = 16  (4 heads * 4 dims = 16)
List<Value> attnOut = new ArrayList<>();   // we'll fill this with 16 numbers

for (int h = 0; h < nHead; h++) {           // one pass per head
    int off = h * headDim;                   // this head owns dims [off, off+4)

    // --- Query, Key, Value for THIS head's slice ---
    List<Value> q = linear(x, attn_wq, off, headDim);  // current token's query
    List<Value> k = linear(x, attn_wk, off, headDim);  // current token's key
    List<Value> v = linear(x, attn_wv, off, headDim);  // current token's value

    keys[li * nHead + h].add(k);             // append -> causal growth
    values[li * nHead + h].add(v);

    // --- Score this query against every stored key ---
    List<Value> scores = new ArrayList<>();
    for (List<Value> pastKey : keys[li * nHead + h]) {
        Value s = dot(q, pastKey).div(Math.sqrt(headDim)); // scaled dot product
        scores.add(s);
    }

    List<Value> weights = softmax(scores);   // turn scores into a distribution

    // --- Blend the stored values by those weights ---
    for (int d = 0; d < headDim; d++) {
        Value acc = new Value(0);
        for (int t = 0; t < weights.size(); t++)
            acc = acc.add(weights.get(t).mul(values[li*nHead+h].get(t).get(d)));
        attnOut.add(acc);                    // append this head's output dims
    }
}

// attnOut now holds 4 heads * 4 dims = 16 numbers, concatenated.
// One final linear layer mixes them back together:
List<Value> out = linear(attnOut, attn_wo);   // attn_wo = output projection

The shape of the data as it flows through:

  1. Input: one 16-number vector for the current token.
  2. Split conceptually into 4 slices of 4.
  3. Each head produces its own 4-number output.
  4. Concatenate the 4 outputs back into 16 numbers.
  5. Run that through attn_wo (the output projection) to get the final 16-number result.
Concatenate, then project. The heads work independently, but the model still needs to combine their findings. Gluing the four 4-number outputs into one 16-number vector and passing it through attn_wo lets the model blend "what the vowel-head found" with "what the start-of-word-head found" into a single coherent update.
But why do we need attn_wo — isn't concatenating the heads enough? Concatenation just lays the four head outputs side by side; at that point the heads have never actually talked to each other — the result is a stack of independent outputs. The output projection is a linear layer that mixes across all 16 concatenated numbers, letting one head's discovery influence how another head's is used. Without it, the heads stay four sealed-off lanes and the model can't combine their insights.

7. The complete attention code, section by section

Here is the whole attention block with numbered markers — simplified from the actual Java port for readability (same math, cleaner variable names; see Chapter 10 for the real code). Read each marker's note underneath. 

List<Value> attention(List<Value> x, int li) {
    List<Value> attnOut = new ArrayList<>();

    for (int h = 0; h < nHead; h++) {                       // (1)
        int off = h * headDim;

        List<Value> q = linear(x, attn_wq, off, headDim);   // (2)
        List<Value> k = linear(x, attn_wk, off, headDim);
        List<Value> v = linear(x, attn_wv, off, headDim);

        keys[li * nHead + h].add(k);                        // (3)
        values[li * nHead + h].add(v);

        List<Value> scores = new ArrayList<>();
        for (List<Value> pk : keys[li * nHead + h])         // (4)
            scores.add(dot(q, pk).div(Math.sqrt(headDim)));

        List<Value> w = softmax(scores);                    // (5)

        for (int d = 0; d < headDim; d++) {                 // (6)
            Value acc = new Value(0);
            for (int t = 0; t < w.size(); t++)
                acc = acc.add(w.get(t).mul(values[li*nHead+h].get(t).get(d)));
            attnOut.add(acc);
        }
    }
    return linear(attnOut, attn_wo);                        // (7)
}
1 Loop over heads. Four independent attention computations, one per head.
2 Project to Q, K, V. Three linear layers turn the input slice into this token's query, key, and value.
3 Append to the growing lists. This is the causal mask in disguise — only past + present tokens are in the list.
4 Scaled scores. Dot the query against every stored key, divided by √(headDim) to keep softmax calm.
5 Softmax. Turn raw scores into attention weights that sum to 1.
6 Weighted sum of values. Blend the stored value vectors using those weights — this is the head's output.
7 Output projection. Concatenated head outputs (16 numbers) pass through attn_wo to produce the final result.
