Embeddings

Beginner   How a computer turns letters into lists of numbers it can actually do math with.

The one-sentence version An embedding is just a lookup table that maps each character to a small list of numbers (a vector), and the model gets to learn what those numbers should be.

The Problem: Computers Only Understand Numbers

Here is an uncomfortable truth that everything in this chapter is built to solve: a neural network cannot add, multiply, or compare the letter 'a'. It has no idea what a letter is. The only thing it can do — the only thing — is arithmetic on numbers: 0.3 × 1.7, 2.1 + 0.4, and so on, millions of times.

So before our GPT can process the name "cat", we have to answer a deceptively simple question:

The core question How do we turn a character like 'c' into number(s) — in a way that lets the model discover that 'c' and 'k' often play similar roles, while 'c' and 'q' do not?

Our model has a fixed alphabet. In the names dataset, that's the 26 lowercase letters a–z plus one special BOS token (Beginning Of Sequence — a marker that says "a name starts here"). That's 27 symbols total. We call this set the vocabulary, and its size is vocabSize = 27.

Tokens, not always letters In our micro-GPT, one token = one character. In ChatGPT, one token is usually a chunk of a word (like " the" or "ing") and the vocabulary has ~100,000 entries. The mechanism is identical; only the vocabulary changes. We use characters because they're easy to see.

First Attempt: One-Hot Encoding

The most obvious way to turn 27 symbols into numbers: give each one a slot. Make a list of 27 zeros, and put a single 1 in the slot that belongs to that character. Everything else stays 0. Because exactly one entry is "hot" (set to 1), this is called one-hot encoding.

It looks like this (showing only the first few slots of each 27-long vector):

CharacterSlotOne-hot vector (27 numbers)
a0[1, 0, 0, 0, 0, …, 0]
b1[0, 1, 0, 0, 0, …, 0]
c2[0, 0, 1, 0, 0, …, 0]
………
z25[0, 0, …, 0, 1, 0]
BOS26[0, 0, …, 0, 0, 1]

Think of it like a row of 27 light switches, one per character. To represent 'c', you flip switch #2 on and leave the other 26 off. It's unambiguous and trivially easy to build. Try it:

Demo 1 — One-hot encoder

Pick a character and watch its 27-number one-hot vector light up. Each cell is one slot in the vector.

Why one-hot is not good enough

One-hot works, but it has a fatal flaw for learning. Measure the "distance" between any two different characters and you always get the same answer. 'a' is exactly as far from 'b' as it is from 'z'. Every pair is equidistant.

The flaw One-hot vectors carry zero information about relationships. The encoding insists every character is an isolated island, equally unrelated to every other. But we want the model to notice that vowels behave alike, or that names often end in certain letters. One-hot makes that impossible to express.

There's a second, more practical problem: size. With 27 characters the vectors are 27 long — annoying but fine. With ChatGPT's 100,000-token vocabulary, every single token would be a vector of 100,000 numbers, almost all zeros. That's enormously wasteful.

But why not just use one-hot encoding and be done with it? Because one-hot encoding insists that every character is equally different from every other one. By its math, 'a' is exactly as far from 'b' as it is from 'z' — there's no notion of "more similar" or "less similar" anywhere in it. But in reality vowels behave more like each other than like consonants, and we want the model to be able to discover and use that. One-hot slams the door on it before training even starts. Learned embeddings leave the door open: similar characters can drift close together as the model learns.

The Real Solution: Learned Embeddings

Here's the key idea, and it's the heart of the chapter. Instead of hardcoding the numbers for each character, we let the model learn them.

We make a much smaller vector for each character — in our code, 16 numbers instead of 27. We call this size nEmbd = 16 (short for "number of embedding dimensions"). At the very start of training, we fill these 16 numbers with small random values. They're meaningless noise. Then, as the model trains on thousands of names, it nudges these numbers up and down, over and over, until they become genuinely useful.

Real-life analogy An embedding is a personality test. Imagine everyone takes a 16-question test and gets 16 scores: [creativity: 8, empathy: 9, logic: 6, …]. Two people with similar scores are "similar personalities" — even though the numbers are a crude simplification of a whole human. Embeddings work the same way: each character gets 16 scores that capture its "personality" in the context of name generation, and the model invents the questions itself.
But why random values? Why not start them all at zero, or some sensible guess? This is worth slowing down for. The honest answer is: we don't know what the best embedding numbers are — discovering them is the entire point of training. So we start from noise, and three properties of that noise matter:
  • Not all the same — if every character started with identical numbers, every character would look identical to the model, and it could never tell 'a' from 'b'. The values must differ so the characters can become different. (This is called "breaking symmetry.")
  • Small — large starting numbers blow up when multiplied together through the network, causing numerical instability. Small values keep the early math well-behaved.
  • Random — randomness is the simplest way to guarantee every character starts slightly different, so each one can evolve down its own path during training.
Random noise, gently shaped by a thousand corrections, is how meaningless numbers turn into meaningful ones.
Analogy: describing people with sliders Imagine describing every person you know with 16 sliders: "how tall", "how loud", "how into sports", and so on. People who are similar end up with similar slider settings, and you can do math on them. An embedding is exactly this, except the model invents the 16 sliders itself — and the sliders rarely map to anything a human would name. They just have to be useful for predicting the next character.
But why 16 dimensions? Why not 4, or 1,000? It's a balancing act. More dimensions give the model more "sliders" — more room to encode subtle distinctions between characters. But every extra dimension is also more numbers to learn, and with our tiny names dataset, too many would let the model overfit (memorize the training names instead of learning general patterns). Too few, and it can't express enough to be useful. 16 is the sweet spot for a toy this size. GPT-4 uses ~12,000 dimensions — but it can afford that richness because it trains on a substantial fraction of the internet, with enough examples to fill all that space meaningfully.

In the Java code, creating this table of learnable numbers is a single line:

// vocabSize = 27 (a-z + BOS), nEmbd = 16
// matrix(rows, cols) makes a grid of learnable numbers, randomly initialized
sd.put("wte", matrix(vocabSize, nEmbd));

wte stands for "word token embedding" (the name comes from Karpathy's original; for us it's really a character embedding). sd is the state dictionary — a big Map holding every learnable group of numbers in the whole model. So this line says: "create a grid with 27 rows and 16 columns, fill it with small random numbers, and remember it under the name wte."

An embedding table is literally a 2D array Forget the fancy word. wte is just a Value[27][16] — 27 rows (one per character), each row holding 16 numbers. Row 0 is BOS's vector, row 1 is 'a''s vector, and so on. (Value is our custom number type that also tracks gradients — that's Chapter 5. For now, read Value as "a double that remembers how to learn.")
Concrete example

Input: the letter 'c', whose token id is 2 (a=0, b=1, c=2, … z=25, BOS=26).

Lookup: grab row 2 of wte — just sd.get("wte")[2], an array index, not a calculation.

Output: [0.05, −0.08, 0.12, 0.03, −0.11, 0.07, 0.14, −0.02, 0.09, −0.06, 0.01, 0.10, −0.04, 0.08, −0.13, 0.05] (16 numbers).

These 16 numbers are the model's current understanding of 'c'. They start as small random noise and get nudged during training until letters that behave alike (like 'c' and 'k') end up with similar rows.

Embedding Lookup: It's Just Array Indexing

Once the table exists, using it is the least mysterious operation in all of deep learning. To get the vector for a character, you convert it to its slot number (its token id) and index into the array. That's it. 

// 'c' is the 4th symbol, so tokenId = 3
int tokenId = 3;

// Look up row 3 of the embedding table — a 16-number vector
Value[] tok_emb = sd.get("wte")[tokenId];

// tok_emb is now the 16 learned numbers that represent 'c'

Compare the two approaches side by side:

One-hot

  • 27 numbers per character
  • Fixed forever, never changes
  • All characters equidistant
  • Encodes identity only

Learned embedding

  • 16 numbers per character
  • Adjusted during training
  • Similar characters end up close
  • Encodes meaning / behavior
Where does one-hot go, then? It doesn't fully vanish — an embedding lookup is mathematically the same as multiplying a one-hot vector by the embedding table. Picking row 3 is what you get if you multiply [0,0,0,1,0,…] by the table. The lookup is just the fast shortcut for that multiplication. Same idea, no wasted zeros.

Worked Example: From One Letter to a Whole Word

Step 1 (simple) — embed 'a'

'a' has token id 1, so we grab row 1 of wte. After training it might look like this (16 numbers, made up here for illustration):

'a' → [ 0.21, −0.05, 0.88, 0.14, −0.62, 0.07, 0.33, −0.41, 0.19, 0.55, −0.12, 0.02, 0.71, −0.28, 0.44, −0.09 ]

That single row of 16 numbers is the model's entire internal idea of "a-ness."

Step 2 (medium) — embed "cat"

A word is just a sequence of characters, so we look up one row per character and stack them. "cat" = c(3), a(1), t(20):

'c' → [ −0.33, 0.41, 0.02, … ]  (16 numbers)
'a' → [  0.21, −0.05, 0.88, … ]  (16 numbers)
't' → [  0.60, 0.17, −0.44, … ]  (16 numbers)

So "cat" becomes a small grid: 3 rows × 16 columns = 48 numbers total. This grid is what actually flows into the rest of the network. Everything downstream — attention, the layers, the final prediction — operates on grids of numbers like this one, never on letters.

Step 3 (advanced) — what these numbers come to mean

In our tiny model the 16 numbers are hard to interpret by eye. But in large models trained on real language, the learned vectors capture genuinely astonishing structure. The most famous example, from a 2013 model called Word2Vec:

vector("king") − vector("man") + vector("woman") ≈ vector("queen")

Read that as arithmetic on meaning. Start at "king", subtract the "man" direction, add the "woman" direction, and you land almost exactly on "queen". The model was never told what royalty or gender are — it discovered that some directions in its embedding space correspond to "gender" and others to "royalty", purely from seeing how words are used. Same trick gives Paris − France + Italy ≈ Rome.

Same idea, bigger numbers Word2Vec used ~300 dimensions; GPT-4 uses ~12,000. Our micro-GPT uses 16, on single characters. The principle is unchanged: meaning becomes geometry. Related things sit near each other; relationships become directions you can add and subtract.

What "close" buys you If 'c' and 'k' sit near each other, then anything the model learns about 'c' partially transfers to 'k' for free. That's the whole payoff: the model doesn't have to learn every character from scratch in isolation — it can generalize across similar ones. One-hot encoding throws this away entirely.

One More Piece: Position Embeddings

There's a gap in everything so far. The embedding for 'a' is the same row of wte whether 'a' is the first letter of a name or the fifth. But position obviously matters — a letter at the start of a name plays a different role than the same letter at the end. The token embedding alone is position-blind.

But why does the model need position information at all? Because the transformer treats each position somewhat independently — it doesn't have a built-in sense of order the way you do when you read left to right. Without a position signal, the model would think an 'a' at the start of a name is the very same thing as an 'a' five letters in. And position matters enormously: the first letter of a name follows completely different statistics than the fifth (names rarely start with certain letters but often end with them). Position embeddings hand the model the "where am I?" information it would otherwise be blind to.

The fix is a second table, built exactly like the first, but indexed by position instead of by character: 

// blockSize = 16 (the most characters we look at at once)
// One learnable 16-number vector PER position slot
sd.put("wpe", matrix(blockSize, nEmbd));

wpe = "word position embedding". Position 0 gets its own 16 numbers, position 1 gets a different 16, … up to position 15. These are learned too — the model figures out how to encode "I am near the start" or "I am near the end."

The two embeddings combine by plain element-wise addition. For a character at a given spot, its final input vector is:

final = token_embedding + position_embedding

That's literally adding two 16-number vectors slot by slot to get one 16-number result. The sum carries both "which character" and "where it sits" in a single bundle — which is exactly what the rest of the network needs.

Real-life analogy Position embeddings are seat numbers at a dinner table. The same person (the same token embedding) behaves differently depending on whether they're sitting at the head of the table (position 0) or in the middle (position 5). Adding the position embedding is like handing each guest a place card — it injects the "where am I sitting?" information that the character identity alone can't carry.
Concrete example

The letter 'e' sitting at position 2 of a name:

Token embedding (from wte, "this is an 'e'"): [0.05, −0.08, 0.12, …]

Position embedding (from wpe, "I'm at slot 2"): [0.01, 0.03, −0.02, …]

Sum (element-wise add): [0.06, −0.05, 0.10, …]

This single 16-number vector now carries both facts at once: which character it is and where it sits. An 'e' at position 0 would start from the same token row but add a different position row, so it ends up as a different vector.

Demo 3 — token + position = final input

Pick a character and a position. The two vectors (shown as colored bars, one bar per dimension) add together into the final input vector. Bars point up for positive numbers, down for negative.

Why add instead of concatenate? You might expect to glue the two vectors into a 32-long one. Addition keeps the size at 16 (cheaper) and works because the model has enough freedom, while learning, to keep "character info" and "position info" in compatible parts of the vector. It's a standard trick straight out of the original Transformer paper.

Recap

  • Computers do math on numbers, not letters — so every character must become a vector first.
  • One-hot gives each of 27 characters a 27-long vector with a single 1. Simple, but every character is equidistant and it carries no relationships.
  • Learned embeddings give each character a short vector (16 numbers, nEmbd) of values the model adjusts during training, so similar characters drift close together.
  • The embedding table wte is literally a Value[27][16] array; a lookup is just sd.get("wte")[tokenId].
  • A second table wpe encodes position. The model's input for each character is token_embedding + position_embedding.
  • This same idea scales straight up to ChatGPT — just more dimensions, more tokens, more data.

Does Input Length Matter for Embeddings?

But wait — does "chocolate" give the same vector as a book about chocolate? No! This is one of the most common misconceptions about embeddings. Let's clear it up.

Same Topic ≠ Same Vector

When you use an embedding API (like Voyage, OpenAI's text-embedding-3, or Google's Gecko), you always get back a fixed-size vector — say, 1024 numbers — regardless of whether your input is one word or an entire book. But the vectors will be different.

Input Output dimension What the vector captures
"chocolate" 1024 numbers The concept of chocolate — food, sweet, brown, cocoa
"I love eating dark chocolate after dinner" 1024 numbers Chocolate + personal preference + dark variety + timing + eating
"A 200-page book about the history of chocolate" 1024 numbers Chocolate + history + agriculture + trade routes + manufacturing + culture + ...

The vectors will be similar (nearby in the 1024-dimensional space) because they're all about chocolate. But they won't be identical — each one captures different nuances.

Concrete example — cosine similarity

If you compute the similarity (cosine similarity) between these vectors:

  • "chocolate" vs "dark chocolate cake recipe" → similarity ≈ 0.82 (high — same topic)
  • "chocolate" vs "quantum physics" → similarity ≈ 0.15 (low — unrelated)
  • "chocolate" vs "a 500-word essay about chocolate" → similarity ≈ 0.75 (high but not identical — the essay has extra context)

Similar meaning → nearby vectors. Different meaning → distant vectors. But never identical unless the input is identical.

How Does the Model Compress a Book into 1024 Numbers?

The embedding model processes your entire input and produces a single summary vector. Think of it like this:

  • Short input ("chocolate"): the 1024 numbers mostly capture the core concept. Many dimensions are "unused" or carry weak signals.
  • Long input (a book about chocolate): the 1024 numbers are packed with information — history, manufacturing, trade, culture — but each detail gets a smaller "share" of the 1024 dimensions. Some information is inevitably lost.
The compression tradeoff A 1024-dimension vector can't perfectly represent a 200-page book. Information IS lost. That's why retrieval systems (like RAG) split documents into chunks (paragraphs or pages), embed each chunk separately, and search across all chunks. Embedding the whole book would "dilute" specific details into a single blurry vector.

How Does This Relate to Our MicroGPT?

Our microGPT uses character-level embeddings: each character ('a', 'b', ..., 'z') gets its own 16-dimensional vector. This is the simplest possible case — one character = one vector, no compression needed.

Production embedding APIs handle token sequences: they process all tokens together through a transformer, then average or pool the outputs into a single vector. The architecture is similar to our GPT — the difference is that production models have been trained specifically to make these summary vectors useful for similarity search.

But why doesn't our microGPT produce sentence embeddings? Because it's a generative model (predicts the next character), not an encoding model (produces a summary vector). Embedding APIs like Voyage use a different training objective: they learn to put similar texts close together and dissimilar texts far apart. Same transformer architecture, different training goal.
